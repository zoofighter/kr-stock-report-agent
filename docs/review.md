# 시스템 설계 검토 의견

## 전체 구조 요약

사용자가 제안한 시스템은 크게 세 단계로 구성됩니다:

1. **데이터 수집 및 인덱싱** — 증권사 리포트 + 뉴스 수집 → RAG DB 저장
2. **분석 및 질의응답** — LLM이 스스로 질문을 생성하고 답변
3. **보고서 자동 작성** — LangGraph 기반 구조화된 리포트 생성

---

## 잘 설계된 부분

### 날짜 기반 가중치
최근 리포트에 가중치를 두는 접근은 금융 도메인에서 매우 중요합니다. 오래된 정보가 최신 판단에 영향을 주는 것을 방지하는 좋은 설계입니다.

### LLM 자기 질문 생성 (Self-Querying RAG)
LLM이 중요한 질문을 스스로 만들고, 그 질문도 RAG에 저장하는 아이디어는 탁월합니다. 이는 향후 유사 리포트 처리 시 "이미 분석된 질문"을 재활용할 수 있어 효율이 높아집니다.

### 뉴스 다중 소스 통합
Naver API + DuckDuckGo + Google News + Naver 블로그를 모두 활용하는 전략은 한국 시장 특성을 잘 반영합니다. 특히 Naver 블로그는 기관 리포트에서 다루지 않는 현장 정보를 보완합니다.

---

## 개선이 필요한 부분

### 1. 뉴스 검색 신뢰도 관리
Naver 블로그는 광고성 콘텐츠나 부정확한 정보가 많습니다. 소스별 신뢰도 점수나 필터링 로직이 없으면 보고서 품질이 저하될 수 있습니다.

**제안:** 소스별 신뢰도 가중치를 별도 관리 (예: 증권사 리포트 > 뉴스 > 블로그 순)

### 2. RAG DB 구조 설계 명확화
현재 `reportdb/`에 무엇이 어떤 형태로 저장되는지 불명확합니다. 리포트 원문, 요약, LLM 질문/답변이 한 공간에 섞이면 검색 정밀도가 떨어집니다.

**제안:** 컬렉션을 분리
- `reports` — 원문 청크
- `summaries` — 요약
- `qa_pairs` — LLM 질문-답변 쌍

### 3. LangGraph 노드 설계
"소제목 또는 키워드를 만들고 그것을 토대로 보고서를 쓴다"는 흐름은 좋지만, 각 섹션 간 문맥 연결(일관성)을 어떻게 유지할지 정의가 필요합니다.

**제안:** LangGraph 상태(State)에 `global_context` 필드를 두고, 이전 섹션의 핵심 논지를 다음 섹션 생성 시 주입

### 4. 보고서 생성 1단계 품질 관리
목차 생성(4~8개)이 랜덤하게 나올 경우, 매번 보고서 구조가 달라져 비교·아카이브가 어렵습니다.

**제안:** 보고서 유형별 목차 템플릿을 사전 정의하고, LLM이 그 안에서 선택/조정하도록 구성

## 추가로 고려할 사항

| 항목 | 내용 |
|------|------|
| **청크 전략** | 증권사 리포트는 섹션 단위로 청크하는 것이 의미 보존에 유리 |
| **캐싱** | 동일 종목 뉴스 중복 검색 방지를 위해 Redis 등 단기 캐시 추가 권장 |
| **평가 루프** | 생성된 보고서의 사실 일치 여부를 LLM이 검증하는 자기 평가(self-critique) 단계 추가 |
| **비용 관리** | LLM 호출이 많아질수록 비용 급증 — 요약/질문 생성에는 소형 모델 혼용 검토 |

---

## 권장 우선순위

1. RAG DB 컬렉션 구조 설계 확정
2. 뉴스 소스별 수집 파이프라인 구현 및 신뢰도 필터 적용
3. LangGraph 플로우 스케치 (노드: 목차 생성 → 섹션별 초안 → 통합 편집)
4. 날짜 가중치 및 소스 가중치 수식 정의

---

전체적으로 실용적이고 현실적인 설계입니다. 핵심은 **RAG 품질**과 **LangGraph 상태 관리**이며, 이 두 가지를 초반에 잘 정의하면 이후 확장이 수월해집니다.
